Holistic Create UX

The real issue is that thread creation is split by internal object type, not user intent.

Today the app asks users to choose between `New Thread` and `New Group Thread` too early, then asks them to discover agents or groups inside separate flows. A better UX starts with one question: “How do you want to begin?” and only then reveals the right setup details.

Current friction Users have to understand the difference between freeform, single-agent, multi-agent, and saved groups before they can start well.
What should feel obvious Start blank, start with an agent, or start with a saved team. Those are user-facing intents.
Design principle Mode and mission should be secondary decisions after choosing the starting shape of the thread.

Option A

Recommended
Replace both dialogs with one unified `Create Thread` sheet. First row is starter type: `Blank`, `Agent`, `Group`. The lower half then adapts to the chosen type.

Why this works
It fixes the discovery problem immediately without fully rewriting the downstream setup sheets.
Minimal disruption. Best if you want a cleaner bridge from today’s flow.

Still a two-step architecture, so less elegant than Option A.

My Recommendation

  • Option A is the best long-term UX. It turns creation into one coherent flow and makes `Blank`, `Agent`, and `Group` feel like peer concepts.
  • Option B is the best bridge if you want faster implementation while preserving most of the current sheet code.
  • Option C is great if you want a more launcher-style product feel, but it optimizes for returning users over guided clarity.
If we’re thinking holistically, I’d choose Option A: one unified `Create Thread` flow, with starter type first, then run mode and mission second.
