# Retrospective Facilitation

## When to Activate

Use at the end of every sprint or milestone, after incidents, or when team morale or delivery quality needs a structured reset. Skip only if the same issues were just retro’d without follow-through—in that case, run a shorter “check actions” session first.

## Process

1. **Create safety** — Start with ground rules: no blame, focus on systems, assume good intent. Use anonymous input (Miro sticky notes, Slido, or a shared doc) if the team is new or tension is high.
2. **Gather data** — Timeline of the period, metrics, incidents, and “moments.” Then collect raw observations: what went well, what was hard, surprises. Do not jump to solutions yet.
3. **Find patterns** — Cluster themes (e.g., “unclear requirements,” “context switching”). Vote on top 2–4 themes to discuss deeply.
4. **Choose experiments** — Pick **1–3** actionable changes only. Each needs an **owner**, a **by date**, and a **signal of success** (e.g., “fewer than 2 scope changes mid-sprint”).
5. **Track prior actions** — Review last retro’s action items at the start every time. Close items done, carry or drop stale ones with explicit agreement.
6. **Rotate facilitation** — Use different formats (Start/Stop/Continue, 4Ls, Sailboat). Rotate facilitator so power dynamics stay balanced.
7. **Close with appreciation** — Each person thanks one teammate for something specific (30–60 seconds).

## Checklist

- [ ] Prior retro actions reviewed at start
- [ ] Data gathered before solutioning
- [ ] 1–3 experiments with owners and dates
- [ ] Facilitation style or facilitator rotated
- [ ] Appreciation round completed

## Tips

If retros feel repetitive, time-box root-cause on one theme only. For remote teams, use a visible board (Miro, FigJam) and keep cameras optional but engagement high via round-robin. Escalate systemic blockers outside the team’s control to leadership with the retro summary, not as venting only.
