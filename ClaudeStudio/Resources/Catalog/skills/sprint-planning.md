# Sprint Planning: Capacity-Based, Goal-Driven Iterations

## When to Activate

Use at the start of each sprint for Scrum teams balancing delivery, tech debt, and unpredictable support work.

## Process

1. **Groom to ready.** Ensure top backlog items have acceptance criteria, designs, and dependencies resolved; spikes flagged when unknowns exceed one day.
2. **Size with history.** Use story points or throughput from last 3–5 sprints; apply a capacity discount for meetings, PTO, and on-call (often 70–85% of ideal dev days).
3. **Protect tech debt.** Allocate explicit bandwidth (e.g., 15–20% capacity) for refactors, flaky tests, and security patches—track it visibly on the board.
4. **Sprint goal and non-goals.** Write a one-line goal customers would recognize; list explicit non-goals to prevent mid-sprint scope creep.
5. **Dependencies and risks.** Surface cross-team needs early; add mitigation tasks (contracts, stubs) instead of hoping for parallel magic.
6. **Definition of Done.** Confirm team agreement: tests, docs, metrics, feature flags, and release notes where applicable. Pull work only that meets DoD assumptions.

## Checklist

- [ ] Backlog top ordered with acceptance criteria
- [ ] Capacity calculated with PTO/on-call
- [ ] Tech debt items budgeted
- [ ] Sprint goal + non-goals published
- [ ] Dependencies have named partners/dates

## Tips

If carryover is chronic, reduce WIP and right-size stories. End planning with a sanity check: “Can we demo the goal on Friday?” If not, descope before coding starts.
