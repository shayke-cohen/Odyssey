# Onboarding Playbook

## When to Activate

Use when hiring engineers or rotating teams; refresh **quarterly** or when tools (SSO, laptop policy, CI) change. Complement (don't duplicate) the repo README—this is the human journey.

## Process

1. **Split by role** — Sections for **application**, **infrastructure/SRE**, **data/ML**—each with tailored access (AWS console, warehouse, notebooks) and first tasks.
2. **Day one** — Laptop setup checklist, SSO apps (Google Workspace, Okta), Slack/Discord channels, PTO policy link, **repo clone**, `gh auth login`, first **hello-world PR** (doc typo or test) merged same week.
3. **Week one** — Architecture tour (recorded + live Q&A), coding standards (`eslint`, `swiftformat`, PR checklist), security training, **on-call shadow** (read-only) if applicable.
4. **Mentors** — Name a **buddy** for culture and a **technical mentor** for code reviews; include calendared 1:1s for the first month.
5. **Systems map** — One page: key services, data stores, and who owns them. Link runbooks and dashboards.
6. **Quarterly refresh** — Assign an owner; diff against new-hire feedback and IT changes.

## Checklist

- [ ] Role-specific paths for app/infra/data
- [ ] Day-one access and first PR within week one
- [ ] Week-one learning goals and shadowing scheduled
- [ ] Buddy and technical mentor assigned
- [ ] Architecture and ownership map linked
- [ ] Last reviewed date in footer

## Tips

Use Notion or internal handbook with checkboxes new hires can self-serve. Avoid dumping 50 links—curate "start here." Survey after 30 days: "What blocked you?" and feed answers back into the guide.
