# Stakeholder Communication

## When to Activate

Use for recurring exec updates, cross-functional syncs, steering committees, or async written status. Activate when decisions are stuck, risks are growing, or stakeholders report “we don’t know what’s going on.”

## Process

1. **Lead with decisions or outcomes** — First lines: what you need from them *today* (approve, choose, unblock) or what shipped and its impact. Put context after, not before.
2. **Use a consistent format** — **Goal** (one line), **Progress** (bullets with dates), **Blockers** (owner + ask), **Asks** (who does what by when). Same skeleton every week builds trust.
3. **Summarize risks with mitigations** — For each risk: likelihood/impact in plain language, what you are doing now, what you need. Avoid fear without a plan.
4. **Record decisions** — After meetings, send a 5-line decision log: decision, alternatives considered, date, decider, link to doc or ADR. Store in Confluence, Notion, or `docs/decisions/`.
5. **Drop jargon** — Define acronyms once; replace internal codenames with customer-visible names when speaking to non-engineers.
6. **Follow up early on slips** — If a commitment slips, notify before the deadline with a new date and cause. Silence erodes credibility faster than bad news.

## Checklist

- [ ] Decision or outcome is in the first screen
- [ ] Goal / progress / blockers / asks structure used
- [ ] Each risk has a mitigation and owner
- [ ] Post-meeting decision record sent within 24h
- [ ] Language is accessible to the audience

## Tips

Attach one chart or screenshot max; link deep dives. For execs, use “so what” lines: “This reduces checkout errors by ~X%.” Calendar a 15-minute optional Q&A after heavy written updates to avoid endless comment threads.
